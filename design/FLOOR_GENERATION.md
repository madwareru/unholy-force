# Алгоритм сборки этажей

Реализация: `src/app/game_stage/floor_generator.rs`

## Обзор

Алгоритм собирает процедурные этажи на основе набора тайловых «частей» (floor parts) 
размером 5×5 и правил их смежности. Он не является классической реализацией Wave Function Collapse, 
но заимствует из неё отдельные структуры данных (модули с множествами допустимых соседей, 
битовые маски вероятностей). Главное отличие — алгоритм не гарантирует полное заполнение карты тайлами; 
если для какой-то позиции не удаётся подобрать подходящую часть, она остаётся заполненной стенами. 
После основной сборки запускается постобработка, которая соединяет разрозненные «острова» проходимости коридорами.

## Входные данные

- **FloorConfig** (`src/game_config/floors.rs`) — конфигурация этажа. Содержит:
  + `floor_variant` — вариант этажа (авторский или сборный).
  + `available_parts` — список ссылок на `FloorPartAdjacencyConfig`, определяющий доступный набор частей и 
    правила смежности.

- **FloorPartAdjacencyConfig** (`src/game_config/floor_part_adjacency.rs`) — для каждой части этажа задаёт 
  списки допустимых соседей по четырём сторонам (север, юг, запад, восток).

- **FloorPartConfig** (`src/game_config/floor_parts.rs`) — тайловая карта части 5×5, содержащая 
  данные пола (`floor_data`), стен (`wall_data`) и дополнительные метаданные (`extra_data`: спавн 
  юнитов, лута, лестницы и т.д.).

## Этапы алгоритма

### 1. Загрузка и ветвление

Функция `generate()` загружает `FloorConfig` из ассетов. Если вариант этажа — **авторский** (`AuthoredFloor`), 
данные возвращаются, как есть (без сборки). Если **сборный** (`GeneratedFloor`) — запускается сборка.

Размер сетки частей зависит от целевого размера этажа:

| Размер этажа | Сетка частей (W×H) |
|--------------|--------------------|
| 15×15        | 3×3                |
| 20×20        | 4×4                |
| 25×25        | 5×5                |
| 30×30        | 6×6                |
| 40×40        | 8×8                |
| 60×60        | 12×12              |
| 80×80        | 16×16              |

Каждая часть — 5×5 тайлов, поэтому размер этажа в тайлах = размер сетки × 5.

### 2. Сбор модулей

Из всех `FloorPartAdjacencyConfig`, перечисленных в `available_parts`, алгоритм 
собирает **уникальные** ссылки на `FloorPartConfig`. Каждой уникальной части 
присваивается числовой индекс и создаётся структура `WfcModule` с пустыми множествами соседей.

Формируются:
- `modules: Vec<WfcModule>` — массив модулей.
- `uuid_mapping: HashMap<ConfigId<FloorPartConfig>, usize>` — отображение UUID части → индекс модуля.
- `module_ids: Vec<ConfigId<FloorPartConfig>>` — обратное отображение индекс → UUID.
- `floor_part_cache` — кэш загруженных конфигураций частей.

### 3. Ограничения для границ этажа

Для каждого модуля проверяются его границы (верхняя/нижняя строка и левый/правый столбец массива `wall_data` 5×5). 
Если все тайлы строки/столбца содержат непустые стены, этот модуль помечается как допустимый для соответствующего 
края этажа:
- `probabilities_north_edge` — модули со сплошной стеной на верхней границе.
- `probabilities_south_edge` — модули со сплошной стеной на нижней границе.
- `probabilities_west_edge` — модули со сплошной стеной на левой границе.
- `probabilities_east_edge` — модули со сплошной стеной на правой границе.

Это гарантирует, что по периметру этажа не будет «дыр» в стенах.

### 4. Построение графа смежности

На основе `FloorPartAdjacencyConfig` алгоритм заполняет множества соседей каждого модуля. 
Для каждой записи смежности (например, «часть A может иметь часть B на севере»):
- В модуль A добавляется `B` как северный сосед.
- В модуль B добавляется `A` как южный сосед (обратная связь).

Аналогично для всех четырёх направлений. Это формирует двусторонний ориентированный граф совместимости.

### 5. Сборка сетки

Этаж инициализируется значениями по умолчанию (все тайлы — стены). 
Затем алгоритм обходит каждую ячейку сетки в порядке **«снаружи внутрь»** 
с помощью итератора `traverse_area_inward()` и для каждой позиции:

1. **Начальное множество кандидатов** — все модули (битовая маска `probabilities`).
2. **Ограничения границ** — если ячейка на северном/южном/западном/восточном краю сетки, множество пересекается с 
   соответствующей маской граничных модулей.
3. **Ограничения соседей** — для каждого уже размещённого соседа (север, юг, запад, восток) из множества кандидатов 
   исключаются модули, не совместимые с этим соседом по соответствующему направлению.
4. **Случайный выбор** — из оставшегося множества кандидатов случайным образом выбирается один модуль. Результат 
   сохраняется в `results[idx]`.
5. **Пустой выбор** — если множество кандидатов оказалось пустым, ячейка остаётся `None` (участок заполняется стенами 
   по умолчанию).

#### Порядок обхода `traverse_area_inward`

Итератор обходит квадратную область спирально от краёв к центру:
1. Четыре угла внешнего кольца.
2. Рёбра внешнего кольца — от углов к центру каждого ребра.
3. Центры рёбер (при нечётной ширине ребра).
4. Переход к следующему вложенному кольцу.
5. Повтор, пока не будет достигнут центр.

Такой порядок обеспечивает максимальное количество ограничений от уже размещённых соседей при обработке каждой 
следующей ячейки.

### 6. Размещение тайлов

Для каждой ячейки сетки, где был выбран модуль, данные соответствующего `FloorPartConfig` (5×5 тайлов пола и стен) 
копируются в итоговый этаж по координатам `[i * 5, j * 5]`.

### 7. Постобработка: соединение островов

Функция `connect_islands()` обеспечивает полную проходимость этажа.

#### 7.1. Определение островов (`get_island_mapping`)

Алгоритм flood-fill с построением DAG:
1. Посимвольный обход этажа слева направо, сверху вниз.
2. Горизонтально смежные проходимые тайлы (стена == `None`) объединяются в один «узел острова».
3. Если над текущим проходимым тайлом есть проходимый тайл из другого узла, создаётся направленная связь снизу вверх. 
   Это формирует DAG без циклов.
4. Корневые узлы (не имеющие входящих связей) распространяют свой «цвет» на всех потомков по графу.
5. Результат — количество уникальных островов и карта `island_mapping`, где каждому тайлу присвоен идентификатор острова.

#### 7.2. Итеративное соединение

Пока островов больше одного:

1. **Поиск лучшего перекрёстка** — для каждой ячейки, заполненной стеной, алгоритм ищет в четырёх кардинальных 
   направлениях ближайшие проходимые тайлы:
   - Если найдены тайлы из разных островов, ячейка-стена становится кандидатом в «перекрёсток».
   - Каждый кандидат характеризуется количеством уникальных связываемых островов и максимальной длиной коридора.

2. **Выбор лучшего** — выбирается кандидат с **наибольшим** числом связываемых островов и **наименьшей** максимальной 
   длиной коридора (приоритет: больше островов → короче коридоры).

3. **Прорубание коридора** — для каждого направления, где найден проходимый тайл другого острова:
   - Все тайлы от перекрёстка до этого тайла превращаются в проходимые (стена → `None`, пол → `Tile`).

4. **Пересчёт** — островная карта пересчитывается заново. Цикл повторяется, пока не останется один остров.

Если подходящий перекрёсток не найден, цикл прерывается с предупреждением в консоль.
